## 一、專題概述

本專題在單一 FPGA 上實作一套**自訂的串列通訊協定**，並以「板內自我迴路（self-loop）」的方式，讓發送端（TX）編碼出的訊號直接送進接收端（RX）解碼，藉此驗證協定的正確性。

在功能正常之後，進一步把設計升級為**真正的雙時脈跨時脈域（Clock Domain Crossing, CDC）架構**：TX 與 RX 分別跑在兩個由 MMCM 產生、且頻率為非整數倍關係的時脈下，並透過 `report_cdc` 驗證跨域路徑的安全性。

接收到的 2-bit 資料會選擇四種展示模式之一，分別驅動板上不同的實體裝置（一般 LED、RGB LED、七段顯示器）。

整體資料流：

```
按鈕 → 防彈跳 → 脈衝產生 → TX 編碼 ──(signal)──→ RX 解碼 → 模式選擇 → 各展示模組
        └──────── clk_tx 時脈域 ────────┘         └──── clk_rx 時脈域 ────┘
```

## 二、通訊協定設計

協定參考 NEC 紅外線遙控的概念自行設計，以**低電位（LOW）的長度**來區分位元 0 與位元 1。

| 符號 | 波形（HIGH / LOW，單位為 TX 時脈週期） | 意義 |
|------|------|------|
| Start  | HIGH 100, LOW 50 | 訊框起始 |
| Logic 0 | HIGH 10, LOW 10 | 位元 0 |
| Logic 1 | HIGH 10, LOW 30 | 位元 1 |
| End | HIGH 10, idle | 訊框結束 |

- 位元 0 與位元 1 的 **HIGH 長度相同**，差別只在 **LOW 的長度**——這是本協定的核心設計，讓接收端只需量測低電位持續時間即可判斷位元值。
- 資料以 **MSB first** 傳送。
- 每個訊框攜帶 2-bit 資料，對應四種模式。

### TX 編碼器

採 6 狀態 FSM（IDLE / START_HIGH / START_LOW / DATA_HIGH / DATA_LOW / END_HIGH）。`signal` 僅在各 `*_HIGH` 狀態拉高。位元索引（bit_index）在進入 DATA 階段時初始化，於 `DATA_LOW → DATA_HIGH` 的轉換點遞減，確保 MSB 先送。四種資料組合（00/01/10/11）皆經模擬驗證為 MSB-first 正確。

### RX 解碼器

- **兩級同步器**：對非同步進來的 `signal` 連打兩級正反器（`s1 <= signal; s2 <= s1;`），後續邏輯一律使用 `s2`，避免亞穩態傳播。
- **邊緣偵測**：以 `s2` 與其前一拍 `prev` 比較，偵測上升／下降緣。
- **6 狀態 FSM**：依序量測 Start、StartLow、各位元的 HIGH/LOW 長度。位元值由 LOW 段的計數值落在哪個區間決定。
- **資料移位**：`data <= {data[0], bit}`，邊收邊移。

## 三、雙時脈跨時脈域（CDC）設計

這是本專題技術上最核心、也最超出基礎範圍的部分。

### 為什麼要做、怎麼做

單一時脈下，TX 與 RX 永遠對齊，跨域問題不會出現。為了製造**真正的**跨時脈域情境，本設計用 **MMCM（Clocking Wizard IP）**產生兩個時脈：

- `clk_tx` = **120 MHz**（TX 端：防彈跳、脈衝產生、TX 編碼）
- `clk_rx` = **80 MHz**（RX 端：RX 解碼、各展示模組、所有輸出 mux）

兩者比例為 **1.5 倍（非整數）**，刻意讓兩個時脈域在相位上互不相關——這正是跨時脈域問題會發生的條件。`signal` 從 120 MHz 域跨進 80 MHz 域，就是真正的 CDC 點。

### MMCM 啟動保護

```verilog
wire sys_rst = rst | ~locked;
```

`locked` 是 MMCM 鎖定完成的旗標。把它與手動 `rst` 做 OR，可確保在 MMCM 尚未穩定輸出時，全系統保持在 reset 狀態，避免用到還沒鎖定的時脈。這是 MMCM 的標準啟動保護寫法。

### 門檻值的重新推導（關鍵理解）

換到雙時脈後，原本的計數門檻**不能直接沿用**，因為 RX 是用**自己的時脈**在數數。同一段實體時間，RX 數到的數值會是：

```
RX_count = 脈衝實際時間 ÷ RX 時脈週期
         = TX_count × (RX_freq / TX_freq)
         = TX_count × (80 / 120)
         = TX_count × 0.667
```

據此重算門檻：

| 符號 | TX 端計數 | 換算後 RX 端計數 |
|------|-----------|------------------|
| Start | 100 | ≈ 66 |
| StartLow | 50 | ≈ 33 |
| Logic 0 | 10 | ≈ 7 |
| Logic 1 | 30 | ≈ 20 |

實作上門檻採**區間**判斷（而非單一值），以容忍時脈抖動（jitter）。此計算由我自行推導，並以 iverilog 模擬驗證跨域 loopback 在四種模式與連續切換下皆正確。

### CDC 安全性驗證

執行 `report_cdc` 後，關鍵證據為：

```
clk_out1_clk_wiz_0 → clk_out2_clk_wiz_0   Safely Timed   Endpoints=1   Safe=1
```

這正是 `signal` 由 TX（120 MHz）跨入 RX（80 MHz）的那條路徑，被工具判定為 **Safely Timed**，代表兩級同步器確實讓這個跨域是安全的。

**為什麼這能證明 CDC 是真的、而且是安全的？** 亞穩態是**機率性**的——只有在訊號變化恰好落在接收正反器的那一小段時間窗（setup/hold 違例區）內才會發生。兩個非整數倍、互不相關的時脈讓這種「邊緣對撞」必然會週期性出現；而兩級同步器的作用，是給第一級正反器一整個時脈週期去把可能的亞穩態收斂掉，使輸出在被使用前已穩定。`report_cdc` 的 `Safely Timed` 即是工具對「此跨域已正確套用同步結構」的確認。

## 四、系統整合架構（COMM_TOP）

頂層的設計重點在於**把「模式編號」與「模式行為」分開處理**：

- **狀態指示（status_led）**：4 顆 LED，以 one-hot 顯示目前是第幾個模式（0001 / 0010 / 0100 / 1000）。單純對 `rx_data` 做 case mux。
- **模式行為**：因為四種行為**驅動的是不同的實體裝置**，無法共用同一組輸出，故拆成三組獨立的輸出 mux：
  - `behavior_led`（4 顆 LED）：跑馬燈（mode 0）／按鈕計數（mode 1）／其餘關閉
  - RGB LED：呼吸燈（mode 2）／其餘關閉
  - 七段顯示器：乘法器結果（mode 3）／其餘關閉（消隱）

這個「狀態用一個 mux、行為依裝置分三個 mux」的架構，是整合階段自己想出來的分法，讓四種完全不同性質的輸出能在同一頂層下乾淨並存。

跨時脈域的劃分：`signal` 是唯一跨域訊號，TX 域產生、RX 域接收，其餘模組各自待在自己的時脈域內，所有模組統一使用 `sys_rst`。

### 各展示模組

- **跑馬燈（Marquee）**：4 顆 LED 輪流亮，每步 0.25 秒。
- **按鈕計數（MC_TOP）**：4-bit 上／下數，可回繞。
- **呼吸燈（Rainbow_Breathing_LED）**：外接 KY-016 RGB 模組於 Pmod JA。
- **乘法器 + 七段（mul_seg）**：4×4 位元乘法，內部用**進位前瞻加法器（Carry-Lookahead Adder）**，結果以七段顯示器掃描輸出。

## 五、開發與除錯歷程

模擬正確、上板卻出錯，是本專題花最多時間、也學到最多的階段。以下幾個 bug 的根因都很有代表性：

1. **RX 切換不了模式（END_LOW 卡死）**
   RX 的 `END_LOW` 狀態原本寫死回自己（`next_state = END_LOW`），導致一個訊框結束後永遠出不來，表現為「換模式沒反應」。修正為偵測到下一個 Start 邊緣時轉回 `START_HIGH`。

2. **資料被綜合最佳化掉**
   資料更新區塊裡混用了阻塞（`=`）與非阻塞（`<=`）賦值，導致綜合工具把暫存器判定為無用而最佳化掉。全部統一為 `<=` 後解決。這也呼應綜合 log 中 `Unused sequential element ... was removed` 的警告。

3. **跑馬燈看起來像恆亮**
   計數門檻一開始設成 `26'd3`，每步只隔 3 個時脈，肉眼看起來就是全亮。改成 `26'd25000000`（0.25 秒）後恢復正常。**教訓**：LED「恆亮」十之八九是計數器太快，不是燈壞。

4. **七段顯示是 16 進位不是 10 進位**
   `3×5=15` 顯示為 `0F`、`9×9=81` 顯示為 `51`，因為直接把二進位值送進七段。若要顯示十進位需另加 BCD 轉換（除 10／取餘，或 double-dabble）。同時修正了七段的位元順序（seg[6]=a … seg[0]=g）與對應的 XDC 腳位。

5. **燒錄 flash 認錯型號**
   Vivado 預設 `s25fl032p`，實際板載為 `mx25l3273f`。程式燒錄時 Vivado 會自動偵測並更正，依正確型號重建 cfgmem 後即燒錄成功。

修正後，四種模式可正常顯示，且**不需按 reset 即可連續切換**，單時脈版完整運作；之後升級的雙時脈版本同樣於實機驗證通過。

- 自訂串列通訊協定（TX 編碼 + RX 解碼）完整實作並上板驗證。

  - 七段顯示器：乘法器結果（mode 3）／其餘關閉（消隱）

這個「狀態用一個 mux、行為依裝置分三個 mux」的架構，是整合階段自己想出來的分法，讓四種完全不同性質的輸出能在同一頂層下乾淨並存。

跨時脈域的劃分：`signal` 是唯一跨域訊號，TX 域產生、RX 域接收，其餘模組各自待在自己的時脈域內，所有模組統一使用 `sys_rst`。

### 各展示模組

- **跑馬燈（Marquee）**：4 顆 LED 輪流亮，每步 0.25 秒。
- **按鈕計數（MC_TOP）**：4-bit 上／下數，可回繞。
- **呼吸燈（Rainbow_Breathing_LED）**：外接 KY-016 RGB 模組於 Pmod JA。
- **乘法器 + 七段（mul_seg）**：4×4 位元乘法，內部用**進位前瞻加法器（Carry-Lookahead Adder）**，結果以七段顯示器掃描輸出。

## 五、開發與除錯歷程

模擬正確、上板卻出錯，是本專題花最多時間、也學到最多的階段。以下幾個 bug 的根因都很有代表性：

1. **RX 切換不了模式（END_LOW 卡死）**
   RX 的 `END_LOW` 狀態原本寫死回自己（`next_state = END_LOW`），導致一個訊框結束後永遠出不來，表現為「換模式沒反應」。修正為偵測到下一個 Start 邊緣時轉回 `START_HIGH`。

2. **資料被綜合最佳化掉**
   資料更新區塊裡混用了阻塞（`=`）與非阻塞（`<=`）賦值，導致綜合工具把暫存器判定為無用而最佳化掉。全部統一為 `<=` 後解決。這也呼應綜合 log 中 `Unused sequential element ... was removed` 的警告。

3. **跑馬燈看起來像恆亮**
   計數門檻一開始設成 `26'd3`，每步只隔 3 個時脈，肉眼看起來就是全亮。改成 `26'd25000000`（0.25 秒）後恢復正常。**教訓**：LED「恆亮」十之八九是計數器太快，不是燈壞。

4. **七段顯示是 16 進位不是 10 進位**
   `3×5=15` 顯示為 `0F`、`9×9=81` 顯示為 `51`，因為直接把二進位值送進七段。若要顯示十進位需另加 BCD 轉換（除 10／取餘，或 double-dabble）。同時修正了七段的位元順序（seg[6]=a … seg[0]=g）與對應的 XDC 腳位。

5. **燒錄 flash 認錯型號**
   Vivado 預設 `s25fl032p`，實際板載為 `mx25l3273f`。程式燒錄時 Vivado 會自動偵測並更正，依正確型號重建 cfgmem 後即燒錄成功。

修正後，四種模式可正常顯示，且**不需按 reset 即可連續切換**，單時脈版完整運作；之後升級的雙時脈版本同樣於實機驗證通過。

## 六、成果與結論

- 自訂串列通訊協定（TX 編碼 + RX 解碼）完整實作並上板驗證。
- 升級為真雙時脈 CDC 架構（120 / 80 MHz，非整數倍），`report_cdc` 確認跨域路徑為 **Safely Timed**。
- 四種展示模式分別驅動一般 LED、RGB LED、七段顯示器，可即時切換。
- 完整時序簽核通過（WNS 為正、hold 無違例），bitstream 產生無 error、無 critical warning。

本專題從協定設計、編解碼 FSM、跨時脈域同步、到多裝置整合與實機除錯，涵蓋了一條完整的數位設計流程。其中雙時脈 CDC 與 `report_cdc` 驗證，是超出基礎邏輯實驗範圍的部分，也是個人投入最多、收穫最大的環節。
